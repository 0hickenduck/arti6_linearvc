# Repo Structure and Harness Review

## 1. Source of Truth
With Trellis initialized, the **source of truth** shifts entirely to the `.trellis/` directory:
- **`.trellis/spec/`**: Persistent source of truth for architectural knowledge, guidelines, and project constraints.
- **`.trellis/tasks/`**: Source of truth for active work, current state, and scoped research findings (in `{TASK_DIR}/research/`).
- **`.trellis/workspace/`**: Developer session history and journals.

*Old files like `PROJECT_STATE.json` and `AGENTS.md` are obsolete and superseded by `.trellis/workflow.md` and `.agents/`.*

## 2. Repo Layout Strategy
**Recommendation: One repo per research demo.**
A multi-project layout adds unnecessary cognitive load and makes specs overly generic. For an experimental demo like ARTI-6 + LinearVC, a focused, single-purpose repository ensures that the `.trellis/spec/` precisely matches the domain of this specific paper/demo.

Recommended structure:
```text
repo/
  .trellis/
  arti6_linearvc_demo/    # Main code
  external/               # Dependencies
  research_notes/         # Legacy notes (archived)
```

## 3. Code Location
For the ARTI-6 LinearVC demo, the code should live in:
**`arti6_linearvc_demo/`** (at the repository root).

*Why:* Keeping it at the root cleanly separates your novel demo code from the frozen `external/arti-6/` codebase and avoids the nested complexity of a `projects/` directory. Template or scaffold scripts generated by AI should initially land in the task directory (e.g. `.trellis/tasks/0x-scaffold/...`) and then be moved to `arti6_linearvc_demo/` upon verification.

## 4. Task Organization
**Recommendation: Separate, granular tasks.**
A single monolithic "local preparation" task defeats the purpose of the Trellis plan-execute-verify loop. The tasks should be split to force the agents to produce concrete artifacts at each step.
